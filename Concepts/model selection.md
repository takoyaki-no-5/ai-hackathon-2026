### モデル選定・発想テク

結論: **用途でモデルとパラメータを分ける**。当日0–15分の読み替えは「発想モード」、Agent本線は「安定モード」。

詳細な製品比較は [[Guides/latest AI|latest AI]]。ここは **APIで触るときの判断だけ**（Cursor IDE → [[cursor models]]）。

### 役割の分け方（本番）

| 用途 | モデル | パラメータ目安 |
|------|--------|----------------|
| テーマ読み替え・アイデア出し | 速い・安い系で十分 | **temperature 高め**（0.8〜1.2） |
| Agent・tool 呼び出し | 安定旗艦 | **temperature 低め**（0〜0.3） |
| Vision（画像入力） | Vision 対応の旗艦 | 低め。解析はブレさせない |
| 画像生成 | Image API 専用 | プロンプトでバリエーション。待ちは UI で隠す |

**reasoning 系**（o 系など）は内部で考えすぎることがある。発想フェーズは **非 reasoning + 高 temperature** の方が「変な案」が出やすい。

### 安定性をわざと下げる（発想）

**temperature** — 次のトークンのランダムさ。上げるほど脱線・意外性が増える。

**top_p** — 候補を上位確率の塊だけに絞る。下げると堅い、上げると散らばる。temperature と **両方いじりすぎない**（どちらか主に触る）。

**実践テク**（Cursor でも API でも同じ）:

- 同じプロンプトを **3回** 回す（温度高め）。真ん中を採用しない
- 「字面の解釈を3つ」「審査員が驚く尖りを5案」と **件数を指定**
- 「普通のSaaS案は禁止」「子供向けに読み替え」など **制約を足す** とブレが意味のある方向に寄る
- 1案に絞る前に **別スレッド** でブレスト → 採用案だけ本線 Agent に流す

### 安定させる（実装・デモ）

- temperature **0〜0.2**、system で役割固定
- tool 引数は **Zod**（[[zod]]）で型を縛る
- 同じ入力でデモ2回目が壊れないよう、**デモ用プロンプト例**を1つ決めておく

### 当日の型（0–15分）

1. テーマ原文を貼る
2. **発想モード**（高 temperature・複数案）で読み替え3〜5案
3. 3分で1案に決める（揉めたら K が切る）
4. 以降の Agent 実装は **安定モード** に切り替え

木の R2 でも同じ手順で練習 → [[study schedule]]

### 触る場所（AI SDK）

```ts
// 発想用ルート（読み替えだけ）
streamText({ model, prompt, temperature: 1.0 })

// Agent 本線
streamText({ model, tools, temperature: 0.2 })
```

モデル名は `stack` の差し替え穴に合わせて1か所で切り替える。
